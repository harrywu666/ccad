# AI 提示词深度分析与优化建议

> 基于对项目全部代码、文档、现有报告的完整审读后撰写
> 日期：2026-03-07

---

## 〇、阅读说明

本报告分为两大部分：**架构层面问题**和**逐阶段提示词优化建议**。

架构层面问题是"地基"，直接影响系统的可维护性和迭代效率；逐阶段建议是"装修"，直接影响审核准确率。建议先解决架构问题，再做提示词调优。

与现有的 `提示词优化建议报告_k2p5.md` 相比，本报告侧重：
1. 指出代码实现与文档设计之间的具体偏差（文档六描述的方案 vs 实际代码）
2. 指出提示词在多个文件中重复/不一致的具体位置
3. 给出可直接对照代码修改的、有具体行号引用的建议

## 一、架构层面的关键问题

### 1.1 提示词双轨并行——最大的维护隐患

系统目前存在**新旧两套尺寸审核实现并存**的问题：

| 实现 | 文件 | 提示词来源 | 是否可配置 |
|------|------|-----------|-----------|
| 新版 | `services/audit/dimension_audit.py` + `prompt_builder.py` | 通过 `ai_prompt_service.resolve_stage_prompts()` | **是** |
| 旧版 | `services/audit_service.py` 第 793-842 行 | 函数内硬编码 `_build_single_sheet_prompt()` / `_build_pair_compare_prompt()` | **否** |

两套提示词**内容相似但不完全一致**，且旧版 `audit_service.py` 中的 `audit_dimensions()` 函数长达 700+ 行，包含了与新版完全重复的图片处理、缓存、并发逻辑。

**风险**：当你在前端「设置页」调优提示词后，旧版调用路径完全不受影响。如果某个入口仍在调用旧版函数，用户看到的效果和预期不一致，排查成本极高。

**建议**：确认旧版 `audit_service.py` 中的 `audit_dimensions()` 是否仍被调用。如果已被新版替代，建议移除或标记为 deprecated，避免后续误用。

### 1.2 三个 Agent 提示词游离于配置体系之外

`kimi_service.py` 中有 3 个 Agent 的提示词完全硬编码，不走 `ai_prompt_service`：

| Agent | 函数 | 行号 | `ai_prompt_service` 中有对应定义？ |
|-------|------|------|---------------------------------|
| 汇总 Agent | `async_summarize_sheet_infos` | 348-359 | **无** |
| 校验 Agent | `async_validate_sheet_catalog_mapping` | 424-436 | **无** |
| 目录识别 | `async_recognize_catalog` | 194-206 | **有**，但没有使用 |

此外，`master_planner_service.py`（行 101-129、303-308）也硬编码了自己的提示词，尽管 `ai_prompt_service` 中已定义了 `master_task_planner` 阶段。

**建议**：
- 将汇总 Agent 和校验 Agent 加入 `PROMPT_STAGE_DEFINITIONS`，使其可在前端配置
- 让 `kimi_service.py` 和 `master_planner_service.py` 统一调用 `resolve_stage_prompts()`
- 保留硬编码作为 fallback（如数据库不可用时），但主路径走配置

### 1.3 目录识别的字段不一致

| 位置 | 输出字段 |
|------|---------|
| `ai_prompt_service.py` catalog_recognition 默认提示词 | `图号`、`图名` |
| `kimi_service.py` async_recognize_catalog 硬编码提示词 | `图号`、`图名`、**`版本`**、**`日期`** |

两处对模型要求的输出字段不同。如果前端用户在「设置页」调优了 catalog_recognition 的提示词，但实际调用走的是 `kimi_service.py` 的硬编码版本，就会出现"改了没效果"的情况。

**建议**：统一输出字段，以包含更多信息的 4 字段版本（图号、图名、版本、日期）为准。

### 1.4 文档六方案与实际实现的差距

文档六（`图纸图像处理与AI数据融合技术方案`）设计了一套非常完善的 Prompt 构造方案，包括：

| 文档六设计的特性 | 实际代码中是否实现 |
|----------------|-----------------|
| 按象限分组数据，每组标注"在图X中查找" | **未实现**——当前只传递扁平 JSON 数组 |
| 象限覆盖范围说明（如"图2覆盖全图左60%宽、上60%高"） | **未实现** |
| 每条数据的象限内局部坐标（`local_x_pct`, `local_y_pct`） | **未使用**——`compact_dimensions()` 只保留 `global_pct`，丢弃了 `in_quadrants` |
| 图片说明段（"图1是全图...图2-5是象限..."） | **部分实现**——提示词中提了"5张图"但未说明哪张是哪个象限 |
| `build_audit_prompt()` 按象限组织的结构化 Prompt | **未使用**——实际使用 `prompt_builder.py` 的简化版本 |

这是最大的提示词质量提升点：文档六的方案比当前实现精密得多，但代码没有落地。

## 二、逐阶段提示词深度分析

### 2.1 目录识别 (catalog_recognition)

**当前提示词**（`ai_prompt_service.py` 行 37-46）：
```
system: 你是室内装饰施工图识别专家，只返回JSON，不要任何解释。
user:   你将收到2张图：第1张是目录表左侧放大图，第2张是全图。
        请以第1张为主提取所有目录条目，结合第2张纠正。
        只返回JSON数组...每条记录字段固定为：图号、图名。
```

**问题与建议**：

1. **缺乏表格结构线索**：施工图目录是表格形式，但提示词没有告诉模型"这是一个表格，行和列是什么"。k2p5 的视觉能力可以自动识别表格，但明确告知可提高准确率和一致性。

2. **缺少编号连续性约束**：施工图目录的图号通常是连续的（如 A1-01, A1-02, A1-03...），提示词应引导模型在输出时检查连续性，发现跳号则标记置信度。

3. **范围编号缺少展开指导**：有些目录行写"A1-01~A1-05"或"A1-01至A1-05"，应指导模型是展开为 5 条还是保留原样。建议保留原样（展开逻辑由后端代码处理更可靠）。

4. **建议增加图片关系说明**：
   ```
   第1张图是目录区域的放大裁剪（左侧约46%宽度），以此图的文字为主。
   第2张图是完整页面缩览，用于确认目录位置和补充上下文。
   如果第1张图中文字清晰，以第1张为准；如果模糊，结合第2张推断。
   ```

5. **建议增加排序和去重要求**：
   ```
   输出时按图号的出现顺序排列（从上到下）。
   如果同一图号出现多次（如修改记录），只保留最新版本。
   ```

### 2.2 单页图纸识别 (sheet_recognition)

**当前提示词**（`ai_prompt_service.py` 行 53-60）：
```
system: 你是施工图识别Agent，只输出JSON。
user:   你将收到同一页施工图的3张裁剪图：第1张左侧，第2张右侧，第3张下方。
        请综合三张图，识别该页唯一的图号和图名。
```

**问题与建议**：

1. **缺少图签位置知识**：施工图的图签（标题栏）几乎总是在**右下角**。当前提示词让模型"综合三张图"，但没有告诉它重点看哪里。建议：
   ```
   识别优先级：
   1. 优先在第2张（右侧）和第3张（下方）的重叠区域（右下角）查找图签
   2. 图签通常包含：图号、图名、比例、设计单位等信息
   3. 如果右下角没有图签，检查左下角（部分图纸的标题栏在左侧）
   ```

2. **裁剪比例信息缺失**：代码中左侧裁 34%、右侧裁 34%（从 66% 开始）、下方裁 34%（从 66% 开始），但提示词没有告诉模型这些裁剪比例。如果模型知道"第1张是全图左侧 34%"，就能更准确地推断空间关系。

3. **缺少对"无图号"情况的处理**：有些图纸确实没有标准图签（如封面、目录页本身），应指导模型在这种情况下如何响应。

### 2.3 汇总 Agent (summarize_sheet_infos)

**当前提示词**（`kimi_service.py` 行 348-358）：
```
system: 你是施工图汇总Agent，只输出JSON。
user:   请对输入的页级识别结果做统一汇总纠偏...
        可修复轻微OCR误差，但不要凭空新增页。
```

**问题与建议**：

1. **未纳入 ai_prompt_service 管理**——这是架构问题，已在 1.2 中提到。

2. **缺少具体的纠偏规则**：什么是"轻微OCR误差"？需要给模型明确的判断标准：
   ```
   纠偏规则：
   - 图号中的 O/0 混淆、I/1 混淆、S/5 混淆属于 OCR 误差，可修正
   - 图号中的分隔符差异（A1-01 vs A1.01 vs A1_01）应统一为项目中最常见的格式
   - 图名中的"布置图/布制图"明显是 OCR 误差，应统一为"布置图"
   - 如果连续页的图号跳号（如 A1-01, A1-03），不能自动补 A1-02
   - 如果某页置信度 < 0.5，在"理由"字段标注"低置信度，建议人工复核"
   ```

3. **缺少上下文推理引导**：汇总 Agent 的核心价值在于能看到"全貌"，应引导它利用这一优势：
   ```
   请利用全局视角进行推理：
   - 如果大部分图号以"A1-"开头，个别识别为"AI-"，很可能是 OCR 误差
   - 如果相邻页的图号连续，中间有一页识别失败，可以推断其图号
   - 如果图号和图名的组合在行业中不合理（如"A1-01"+"立面详图"），标注异常
   ```

### 2.4 校验 Agent (validate_sheet_catalog_mapping)

**当前提示词**（`kimi_service.py` 行 424-436）：
```
system: 你是施工图匹配校验Agent，只输出JSON。
user:   请将 pages 与 catalog 做一对一匹配...
        允许轻微OCR误差，优先图号，其次图名。
```

**问题与建议**：

1. **未纳入 ai_prompt_service 管理**。

2. **匹配策略过于简单**："优先图号，其次图名"缺乏对边缘情况的覆盖：
   ```
   匹配策略（按优先级）：
   1. 图号精确匹配（最高优先级）
   2. 图号模糊匹配（忽略分隔符差异、O/0混淆等）
   3. 图名 + 序号综合匹配（图号完全不同但图名高度相似时）
   4. 位置推断匹配（相邻页已匹配，根据目录顺序推断当前页）
   
   约束条件：
   - 每个 page 最多匹配一个 catalog 条目
   - 每个 catalog 条目最多被一个 page 匹配
   - 如果一个 page 无法匹配任何 catalog，对应字段留空，置信度设 0.0
   - 如果一个 catalog 没有被任何 page 匹配，不需要输出（这在输入数据中体现为缺页）
   ```

3. **缺少冲突处理**：多个 page 可能争抢同一个 catalog 条目，应指导模型如何取舍：
   ```
   冲突处理：
   如果多个 page 都可能匹配同一个 catalog 条目，选择置信度最高的那个。
   被淘汰的 page 尝试匹配其他 catalog 条目，如果没有合适的，字段留空。
   ```

### 2.5 总控任务规划 (master_task_planner)

**问题与建议**：

1. **`ai_prompt_service` 中有定义，但 `master_planner_service.py` 没有使用它**（行 101-129 硬编码、行 303-308 硬编码 system_prompt）。两处的 system_prompt 甚至不同：
   - `ai_prompt_service` 定义的：`"你是 CAD 审图系统的总控调度 Agent。你要生成可执行任务图（DAG）的任务节点列表。"`
   - `master_planner_service.py` 实际使用的：`"你是施工图审图总控 Agent，负责把输入图纸关系生成可执行任务图。必须只返回 JSON。"`

2. **prompt 过于"命令式"，缺少"推理引导"**：当前的 7 条"必须遵守"规则本质上是在限制模型的输出空间。但问题是——这些限制在代码中已经有 `_normalize_task_item()` 做后校验了（行 143-201）。双重限制的结果是：prompt 太长但模型仍可能违反规则，而代码会兜底纠正。

   建议简化 prompt 中的硬约束（代码已经兜底），转而增加"如何做出好的规划决策"的推理引导：
   ```
   你的目标是生成一套高效的审核计划。思考以下问题：
   - 哪些图纸被引用最多？它们是核心图纸，应该优先审核
   - 索引多的图纸意味着跨图关系复杂，index 审核优先级应更高
   - 平面图是施工图的"锚"，平面图相关的任务优先级最高
   ```

3. **覆盖性校验建议移到代码层**：当前 prompt 要求"对每个有效 edge，至少生成 1 条 dimension 和 1 条 material 任务"，代码中也有覆盖性校验（行 354-385）。建议从 prompt 中移除覆盖性要求，完全依赖代码层校验——因为代码校验 100% 可靠，而模型理解"覆盖性"可能产生冗余或遗漏。

### 2.6 尺寸单图语义分析 (dimension_single_sheet)

**当前提示词**（`ai_prompt_service.py` 行 109-119）：
```
system: 你是专业施工图审核专家，擅长结合全图与象限图做尺寸语义解析。只返回JSON数组，不要解释。
user:   对图纸（{{sheet_no}} {{sheet_name}}）做尺寸语义分析。
        你将收到5张图：图1全图（带网格），图2-5为四个高清象限（20%重叠）。
        请按"先定位，再理解语义"执行：先在图1找网格位置，再去对应象限图确认。
        以下是DWG提取的精确尺寸数据（无需重新OCR）：{{dims_compact_json}}
```

**这是系统中最重要的提示词，也是改进空间最大的。**

**问题 1：5 张图的对应关系不明确**

提示词只说"图2-5为四个高清象限"，但没有说明：
- 图2 = 左上象限
- 图3 = 右上象限
- 图4 = 左下象限
- 图5 = 右下象限

代码中 `_execute_sheet_jobs()`（dimension_audit.py 行 386-393）按 `[full, top_left, top_right, bottom_left, bottom_right]` 顺序传入，但提示词里模型不知道图2是左上。文档六明确定义了这个对应关系，但没有写入提示词。

**建议补充：**
```
你将收到5张图：
- 图1：完整施工图全图（叠加了 A-X 列 × 1-17 行坐标网格）
- 图2：左上象限高清图（覆盖全图 X:0-60%, Y:0-60%）
- 图3：右上象限高清图（覆盖全图 X:40-100%, Y:0-60%）
- 图4：左下象限高清图（覆盖全图 X:0-60%, Y:40-100%）
- 图5：右下象限高清图（覆盖全图 X:40-100%, Y:40-100%）
相邻象限有20%的内容重叠，确保边界处的标注不被截断。
```

**问题 2：数据中有定位信息但提示词不解释**

`compact_dimensions()` 输出的每条尺寸包含 `global_pct`（全图百分比坐标）、`grid`（棋盘格坐标）、`in_quadrants`（所属象限），但提示词没有告诉模型这些字段的含义和用法。

文档六中设计了非常详细的坐标说明段：
```
坐标说明：
  全图位置：棋盘格坐标，如「F11」表示F列第11行
  象限内位置：在该象限图中的百分比位置，左上角为原点(0%,0%)
  重叠区：标注「重叠区，多象限可见」的数据点同时出现在多张象限图中
```

但实际提示词中完全没有。

**建议补充**（在 dims_compact_json 之前）：
```
以下是DWG提取的精确尺寸数据（数值100%准确，无需重新OCR）。
每条数据包含：
- id：标注唯一ID
- value：精确数值（mm）
- grid：在图1棋盘格中的位置（如"F11"表示F列第11行）
- global_pct：在全图中的百分比位置（x=0是最左，y=0是最上）
- in_quadrants：该标注在哪些象限图中可见（重叠区域的标注会出现在多个象限中）

分析步骤：
1. 在图1的棋盘格中定位每条尺寸的大致区域
2. 根据 in_quadrants 找到对应的高清象限图（图2-5）
3. 在高清象限图中确认该尺寸标注的上下文（标注的是哪面墙、哪个空间）
4. 结合空间上下文给出语义解析
```

**问题 3：尺寸分类指导缺失**

施工图中的尺寸有明确的分类（总尺寸、分段尺寸、定位尺寸、细部尺寸），但提示词没有告诉模型这些分类。这导致模型在 `dim_type` 字段的输出不稳定。

**建议增加尺寸分类说明**：
```
尺寸类型说明（填入 dim_type 字段）：
- "总尺寸"：房间或空间的总长度/宽度，通常是最外侧标注链的最大值
- "分段尺寸"：构成总尺寸的各部分尺寸，在总尺寸标注链的内侧
- "定位尺寸"：构件相对于轴线或参考面的距离
- "细部尺寸"：局部详图中的精确尺寸，通常数值较小
- "标高"：竖向标注，单位通常是m而非mm
```

**问题 4：120 条截断无提示**

`compact_dimensions()` 限制最多 120 条尺寸（行 14：`for item in dims[:120]`），但提示词没有告诉模型数据可能被截断。如果一张复杂平面图有 200+ 条尺寸，模型拿到 120 条后不知道还有遗漏的。

**建议**：在 prompt 中动态说明：`"以下数据共 {{total_count}} 条，因长度限制仅展示前 120 条。"`

### 2.7 尺寸双图对比 (dimension_pair_compare)

**当前提示词**（`ai_prompt_service.py` 行 132-159）：
```
user:   请对比两张图的尺寸语义列表，找出同一构件/空间的尺寸不一致项。
        你必须按流程执行：先定位 -> 再配对 -> 再核对。
        规则：
        1) 先根据 semantic/component/grid 选候选，不要直接猜。
        2) 只有定位证据充分时才输出问题；证据不足就不要输出。
        3) 输出必须带证据字段，便于人工复核。
```

**问题 1：缺少差异容忍度指导**

提示词说"找出尺寸不一致项"但没有定义什么算"不一致"。建议参考工程实践：
```
差异判断标准：
- 差值 = 0：完全一致，不输出
- 0 < 差值 ≤ 3mm：工程精度内偏差，不输出
- 3mm < 差值 ≤ 10mm：可能的标注精度差异，输出但 confidence 标低（0.3-0.5）
- 差值 > 10mm：大概率存在不一致，输出且 confidence 标高（0.7+）

特别注意：
- 单位换算问题：如果一张图标 2400mm，另一张标 2.4m，这是同一尺寸，不是不一致
- 总尺寸 vs 分段之和：A图总尺寸 5000mm = B图分段 2400 + 2600mm，不是不一致
- 比例差异：平面图 1:50 和详图 1:10 标注的是不同精度的同一构件，允许轻微差异
```

**问题 2：输出字段中英混杂**

当前输出格式中，部分字段用中文（`位置描述`、`A图号`、`B图号`、`A值`、`B值`、`差值`），部分用英文（`source_grid`、`target_grid`、`source_dim_id`、`confidence`、`description`、`evidence`）。

这种混杂会增加后端解析的复杂度——`result_parser.py` 需要同时处理多种可能的 key 名。建议统一为全英文，或全中文。考虑到代码中已使用英文字段，建议统一为英文：
```json
{
  "location": "主卧东侧墙体净宽",
  "source_sheet_no": "A1-01",
  "target_sheet_no": "A2-03",
  "source_value": 3600,
  "target_value": 3400,
  "diff": 200,
  "source_grid": "C5",
  "target_grid": "D2",
  "source_dim_id": "dim_042",
  "target_dim_id": "dim_128",
  "index_hint": "索引①",
  "confidence": 0.88,
  "description": "...",
  "evidence": { ... }
}
```

**问题 3：没有图片输入但需要空间推理**

双图对比阶段只接收两组语义 JSON（没有图片），但需要模型理解空间对应关系。这意味着单图分析阶段输出的 `semantic`、`component`、`grid` 等字段的质量直接决定对比效果。

建议在提示词中明确说明数据来源和信任度：
```
以下是两张图各自经过"单图语义分析"后的结构化结果。
每条数据的 semantic/component/grid 字段是上一步 AI 分析的产物，可能存在误差。
在配对时，请：
- 优先使用 grid 字段做空间匹配（因为 grid 来自精确的 CAD 坐标）
- 其次使用 component 做语义匹配
- semantic 字段仅作参考（可能有描述差异但指向同一构件）
```

### 2.8 温度参数的不一致

| 阶段 | `ai_prompt_service` 中的温度 | 实际代码中使用的温度 |

| catalog_recognition | 未定义 | `call_kimi` 默认 0.1 |

## 三、提示词质量的系统性改进方向

### 3.1 System Prompt 过于简单

当前所有阶段的 system_prompt 都是一行话（如"你是室内装饰施工图识别专家，只返回JSON，不要任何解释"）。

k2p5 的 system_prompt 是注入领域知识的最佳位置。建议将 system_prompt 扩展为包含：
- 角色定位（你是谁，擅长什么）
- 领域知识（施工图的行业规范、标注习惯）
- 行为约束（输出格式、不做什么）

以尺寸单图语义分析为例：
```
你是专业的室内装饰施工图审核专家。

你擅长：
- 识别施工图中的尺寸标注及其空间含义
- 理解CAD棋盘格坐标系（A-X列 × 1-17行）
- 将尺寸标注与建筑构件（墙体、门窗、空间分隔）关联

施工图基本知识：
- 平面图通常是 1:50 或 1:100 比例
- 尺寸标注分为总尺寸、分段尺寸、定位尺寸、细部尺寸
- 尺寸链：从外到内依次为总尺寸→分段尺寸→细部尺寸
- 轴线通常标注为：横向①②③...、纵向A B C...

输出要求：
- 严格返回JSON数组格式
- 不要markdown代码块
- 不要任何解释性文字
```

### 3.2 缺少"不确定时怎么办"的指导

当前所有提示词都假设模型能正确处理输入。但实际中存在：
- 图片模糊或被遮挡
- CAD 数据与图片不匹配（如 DWG 版本不同）
- 图纸布局非常规（如多个视图叠加在一页）

建议在每个阶段增加"异常处理"段：
```
如果遇到以下情况：
- 图片不清晰导致无法确认标注内容：将该条数据的 confidence 设为 0，evidence.why 填"图片模糊"
- CAD数据中的尺寸在图片上无法对应：将 confidence 设为 0.3，evidence.why 填"CAD数据与图片不匹配"
- 图纸布局非标准（如多视图共页）：尽力分析，在 evidence.why 中标注"非标准布局"
```

### 3.3 Few-shot 示例的精准使用

现有报告建议增加 Few-shot 示例，这是正确的方向，但需要注意**示例的选择必须精准**。

建议为每个阶段维护 2-3 个典型 case 的输入输出对，存储在数据库中，在 `resolve_stage_prompts()` 时动态注入。选择标准：
- 1 个"标准情况"：最常见的输入输出
- 1 个"边缘情况"：模型容易出错的场景
- 1 个"无结果情况"：输入正常但应该输出空结果

这比在提示词中硬编码示例更灵活，也可以随着项目经验积累而丰富。

### 3.4 提示词长度与 token 效率

尺寸审核阶段的提示词 + 数据 JSON 可能非常长（120 条尺寸数据序列化后轻松超过 10K tokens）。建议：

1. **压缩数据格式**：当前 `compact_dimensions()` 已做了字段精简，但可以进一步压缩——比如 `global_pct` 可以从 `{"x": 38.2, "y": 64.5}` 简化为 `[38.2, 64.5]`

2. **分批处理超大图纸**：如果尺寸数量超过 120 条（已被截断），可以考虑分象限提交，每次只提交一个象限的数据和对应图片

3. **提示词模板 vs 数据分离**：固定的指令部分（提示词模板）和变化的数据部分（JSON）应该有明确的分隔标记，方便模型区分"这是指令"和"这是数据"
