# 误报反馈自循环系统设计

**目标**

把当前“误报留言框”升级成一套真正可交互、可判断、可记录、可学习的反馈闭环：

- 用户不是只留一句备注
- 反馈会提交给专门的反馈判定 Agent
- Agent 能结合现场证据回复用户、追问用户、给出结论
- 系统能把过程和结论结构化记录下来
- 只有通过门禁的反馈才进入学习层，避免经验库学坏

---

## 一句话结论

这次要升级的根因，不是弹窗太小，也不是备注不够长。

真正要改的是：

**把“误报反馈”从一个字段，升级成一条反馈会话任务。**

也就是：

- 现在：用户给一条 `feedback_note`
- 以后：用户向“反馈判定 Agent”发起一条会话

Agent 处理完后，系统再决定：

- 这条是否真是误报
- 是否还需要追问
- 是否值得进入学习层
- 是否只做记录、不做学习
- 是否需要转人工

一句大白话：

**不是往系统里塞一张纸条，而是把这条问题交给一个会处理反馈的 Agent。**

---

## 这次做什么，不做什么

### 做什么

- 定义反馈会话层，而不是继续只存 `feedback_status`
- 定义反馈判定 Agent 的职责
- 定义多轮交互状态机
- 定义学习门禁，控制什么能进经验层
- 定义前端从“小弹窗”升级成“会话面板”的形态
- 定义和现有 `feedback_samples` / `feedback_runtime_service` 的衔接方式

### 不做什么

- 这份文档不直接改业务代码
- 这份文档不推翻现有审图主流程
- 这份文档不让用户反馈直接改写审查结论
- 这份文档不让经验库直接强改各审查 Agent 的最终输出
- 这份文档不一开始就做全自动学习和全自动阈值回写

---

## 为什么当前形态不够

现在系统已经做了两件事：

1. 前端允许用户把某条问题标成 `incorrect`
2. 后端把这条反馈同步成 `feedback_samples`

这一步解决的是“能收集误报”。

但还没解决 4 个关键问题：

1. **没有交互**
   用户提交后，系统不会解释自己怎么判断，也不会追问。

2. **没有判定层**
   用户一句“这是误报”，现在更像直接变成样本候选，中间缺少裁判。

3. **没有学习门禁**
   反馈是否值得学习，目前主要靠后续人工整理，不够实时，也不够结构化。

4. **没有反馈过程记录**
   现在主要记录结果，没有完整保存“用户说了什么、Agent 怎么回、为何采纳/不采纳”。

所以它更像：

- 留言板
- 样本入口

而不是：

- 反馈处理系统
- 自循环学习系统

---

## 总体设计

我建议把系统拆成 4 层：

1. **反馈会话层**
2. **反馈判定 Agent**
3. **学习门禁层**
4. **经验运行时层**

关系如下：

```text
用户 -> 反馈会话层 -> 反馈判定 Agent -> 学习门禁层 -> 经验运行时层
                            |                |
                            |                -> 进入样本池 / 拒绝学习 / 转人工
                            -> 回复用户 / 继续追问
```

### 1. 反馈会话层

这是用户能看到的部分。

职责：

- 承接“这条是误报”的发起动作
- 展示问题上下文、命中规则、图纸定位、当前证据
- 承载用户与 Agent 的多轮消息
- 展示当前处理状态和最终结论

它不只是输入框，而是一条和具体 `audit_result` 绑定的反馈会话。

### 2. 反馈判定 Agent

这是这次升级的核心角色。

它不是“训练 Agent”，而是“反馈裁判 Agent”。

职责：

- 读取当前问题上下文
- 读取已有规则、证据、图纸定位、置信度、来源 Agent
- 读取历史类似误报经验
- 判断用户反馈是否站得住
- 决定是否追问用户
- 产出结构化结论
- 给出用户可读的回复

它的重点是：

**先把这条反馈处理清楚，再决定要不要学习。**

### 3. 学习门禁层

这一层专门负责“别乱学”。

职责：

- 判断本次反馈信息是否足够
- 判断证据是否充分
- 判断是否有相似样本支撑
- 判断该模式是否具有复用价值
- 决定是否进入学习池

一句大白话：

**不是用户说了就学，而是先过门禁。**

### 4. 经验运行时层

这一层对接已有 `feedback_runtime_service` 思路，但输入源升级为“经过判定和门禁筛选的结构化经验”。

职责：

- 给各审查 Agent 提供柔性的经验提示
- 影响任务优先级、补图倾向、最低置信门槛、结果状态建议
- 不直接硬改最终判定

---

## 角色分工

### 用户

用户负责：

- 提出“这条像误报”
- 补充场景信息
- 回答 Agent 的追问
- 必要时确认项目内叫法、别名、习惯写法

用户不负责：

- 直接定义规则
- 直接决定系统一定学习

### 反馈判定 Agent

反馈判定 Agent 负责：

- 面向用户对话
- 解释当前系统为什么报错
- 判断是否需要补信息
- 判断是否成立为误报
- 判断是否值得进入学习层
- 给出理由和下一步动作

它不负责：

- 直接改审图主流程代码
- 直接写死规则
- 绕过门禁直接加入强经验

### 人工整理者

人工整理者依旧保留，但职责变化：

- 从“逐条看原始留言”
- 变成“审阅 Agent 已经结构化处理过的候选经验”

这样人工介入的成本会低很多。

## 反馈会话模型

### 为什么必须有会话层

如果没有会话层，系统只能做两件事：

- 收一句话
- 存一个状态

这会导致：

- Agent 没机会追问
- 用户看不到系统在想什么
- 无法区分“反馈已收到”和“反馈已判定”
- 无法保存反馈处理过程

所以反馈对象必须从“备注字段”变成“会话实体”。

### 建议的核心对象

#### `feedback_threads`

表示一条反馈会话。

建议字段：

- `id`
- `project_id`
- `audit_result_id`
- `audit_version`
- `status`
- `learning_decision`
- `agent_decision`
- `agent_confidence`
- `opened_by`
- `source_agent`
- `rule_id`
- `issue_type`
- `summary`
- `resolution_reason`
- `escalation_reason`
- `created_at`
- `updated_at`
- `closed_at`

#### `feedback_messages`

表示一条会话中的消息。

建议字段：

- `id`
- `thread_id`
- `role` (`user` / `agent` / `system`)
- `message_type` (`claim` / `question` / `answer` / `decision` / `note`)
- `content`
- `structured_json`
- `created_at`

#### `feedback_learning_records`

表示学习门禁后的结构化记录。

建议字段：

- `id`
- `thread_id`
- `project_id`
- `audit_result_id`
- `decision` (`accepted` / `rejected` / `human_review`)
- `reason_code`
- `reason_text`
- `evidence_score`
- `similar_case_count`
- `reusability_score`
- `suggested_intervention_level`
- `snapshot_json`
- `created_at`

### 和现有表的关系

现有：

- `audit_results.feedback_status`
- `audit_results.feedback_note`
- `feedback_samples`

建议保留，但角色调整为：

- `audit_results.feedback_status`
  - 用作“这条问题有人发起过误报反馈”的轻量视图状态
  - 不再承载完整流程语义

- `audit_results.feedback_note`
  - 兼容旧界面
  - 可用作会话首条用户消息的摘要镜像

- `feedback_samples`
  - 不再直接吃原始留言
  - 改为主要接收 `feedback_learning_records` 中被接受的结构化快照

一句大白话：

**旧字段留着做兼容，真正的新事实来源是会话表。**

## 状态机

第一版不要搞太复杂，我建议用下面这组状态。

### 会话状态 `status`

- `open`
  - 用户刚发起，待 Agent 处理
- `agent_reviewing`
  - Agent 正在看上下文和历史经验
- `agent_needs_user_input`
  - Agent 认为还缺关键信息，正在追问
- `resolved_incorrect`
  - Agent 判断这条大概率是误报
- `resolved_not_incorrect`
  - Agent 判断当前证据下不能认定为误报
- `escalated_to_human`
  - 转人工复核
- `closed`
  - 本次会话收口

### 学习决策 `learning_decision`

- `pending`
- `accepted_for_learning`
- `rejected_for_learning`
- `needs_human_review`
- `record_only`

### 推荐状态流转

```text
open
 -> agent_reviewing
   -> agent_needs_user_input
      -> agent_reviewing
   -> resolved_incorrect
      -> accepted_for_learning / record_only / needs_human_review
      -> closed
   -> resolved_not_incorrect
      -> rejected_for_learning / closed
   -> escalated_to_human
      -> needs_human_review / closed
```

### 为什么拆两套状态

因为“是不是误报”和“要不要学习”不是一回事。

例如：

- 某条问题可以被判定为误报
- 但它只是一次性的项目特殊写法
- 这种情况应该 `resolved_incorrect + record_only`
- 不应该强行 `accepted_for_learning`

这层拆开后，系统才不会乱学。

## 反馈判定 Agent 的工作方式

### 输入

反馈判定 Agent 每次判断至少要拿到这些上下文：

- 当前 `audit_result` 全量结构化信息
- 图纸定位和证据摘要
- 触发的规则 `rule_id`
- 来源审查 Agent `source_agent`
- 当前置信度和状态
- 当前项目里已采纳的类似经验
- 用户本轮消息和历史消息

### 第一版相似案例怎么查

这块第一版必须写死边界，不要让实现时自由发明检索逻辑。

建议规则：

- 只查已采纳的结构化学习记录
- 只看同一 `rule_id + issue_type`
- 按 `evidence_score` 降序，再按时间倒序
- 最多取最近 `3` 条
- 第一版不做向量检索，不做语义召回

这样做的原因很简单：

- 足够稳
- 足够便宜
- 足够可解释

等最小闭环跑顺后，再考虑更复杂的相似性检索。

### 输出

不要只输出一段自然语言，必须有结构化结果。

建议结构：

```python
class FeedbackAgentDecision(BaseModel):
    status: Literal[
        "resolved_incorrect",
        "resolved_not_incorrect",
        "agent_needs_user_input",
        "escalated_to_human",
    ]
    user_reply: str
    summary: str
    confidence: float
    reason_codes: list[str]
    needs_learning_gate: bool
    suggested_learning_decision: Literal[
        "pending",
        "accepted_for_learning",
        "rejected_for_learning",
        "record_only",
        "needs_human_review",
    ]
    follow_up_question: str | None
    evidence_gaps: list[str]
```

### 它应该优先做的判断

1. 这条反馈是否可能成立
2. 当前缺不缺信息
3. 缺的信息是不是用户能补
4. 这是不是项目私有习惯，而非通用经验
5. 是否值得进入学习门禁

### 它不应该直接做的事

- 不直接把结论写成运行时强规则
- 不直接把 `feedback_samples` 标成 `accepted`
- 不直接改动审图原始结果

## 学习门禁

这是这套系统最重要的安全层。

### 为什么必须有门禁

如果没有门禁，系统会出现两个问题：

1. 用户主观意见会直接污染经验库
2. Agent 的一次误判会被放大成长期偏差

所以学习门禁必须独立存在，不能混在 Agent 回复逻辑里。

### 第一版建议看 4 个维度

#### 1. 信息完整度

至少看：

- 用户是否给了具体原因
- Agent 是否拿到了足够上下文
- 是否还有关键证据缺失

#### 2. 证据充分度

至少看：

- 当前定位是否清楚
- 规则触发原因是否明确
- 是否能从现有图纸中找到支持或反证

#### 3. 相似案例支持度

至少看：

- 本项目内有没有类似误报
- 全局有没有相同 `rule_id + source_agent + 场景` 的已采纳样本

#### 4. 可复用价值

至少看：

- 这是不是通用模式
- 还是只在这个项目里成立的特例

### 第一版门禁结果

- `accepted_for_learning`
  - 信息足够、证据较强、具备复用性
- `record_only`
  - 反馈成立，但偏项目特例，只记录不学习
- `rejected_for_learning`
  - 反馈不成立，或信息太弱
- `needs_human_review`
  - 很像重要经验，但系统还不够确定

### 和现有整理页的关系

现有误报调优页不废弃，但角色升级为：

- 看结构化候选记录
- 做最终采纳/拒绝
- 导出高质量样本

也就是把人工工作从“初筛”变成“复核”。

---

## 经验运行时层如何接入

你们现在已经有 `feedback_runtime_service`，方向是对的。

但以后它的输入不要直接来自原始留言，而应该来自：

- `accepted_for_learning` 的结构化记录
- 聚合后的运行时经验视图

### 保持放权式接口

这一点沿用你们已有思路，不要改回硬补丁系统。

也就是经验层继续输出类似：

- `false_positive_rate`
- `confidence_floor`
- `needs_secondary_review`
- `severity_override`
- `intervention_level`
- `reason_template`

经验层影响的是：

- 任务优先级
- 补图倾向
- 最低置信门槛
- `confirmed / suspected / needs_review` 的落点建议

不影响的是：

- 审查 Agent 的最终硬结论

一句大白话：

**经验层给方向，不替 Agent 下最终判决。**

## 前端界面建议

### 不建议继续用小弹窗

小弹窗适合一次性确认，不适合多轮交互。

如果继续沿用小弹窗，会有几个问题：

- 上下文装不下
- 对话装不下
- 看不到状态流转
- 用户感知不到 Agent 在处理

### 推荐形态：反馈会话抽屉

入口仍然保留小旗子，降低学习成本。

点击后，打开右侧抽屉或详情面板。

建议分 3 块：

#### 1. 问题上下文区

展示：

- 问题描述
- 规则命中原因
- 来源 Agent
- 当前置信度
- 图纸定位和证据图

#### 2. 会话区

展示：

- 用户消息
- Agent 回复
- Agent 追问
- 用户补充
- 最终结论

#### 3. 处理结果区

展示：

- 当前状态
- 是否已确认误报
- 是否进入学习
- 原因说明
- 记录时间

### 第一版最重要的体验变化

用户点完“误报”后，不能只是静默成功。

必须让 Agent 立刻回一句话。

例如：

- “我先看了这条问题的规则和定位，当前更像是图号别名导致的误判。我先按误报候选处理；如果你愿意，可以补充这张图在项目里的常用叫法，我再判断是否写入经验库。”

这句话的价值是：

- 用户知道系统在处理
- 用户知道系统为什么这么想
- 用户知道下一步还能继续协作

## 记录与审计

这套系统必须能回答下面这些问题：

- 这条误报是谁发起的
- Agent 看了哪些上下文
- Agent 为什么认可或不认可
- Agent 为什么决定学习或不学习
- 最终是哪条经验进入了运行时层

所以建议至少保留 3 类记录：

1. **会话消息记录**
2. **结构化判定记录**
3. **学习门禁记录**
4. **相似案例命中记录**

这样后面排查“为什么系统学坏了”时，才能追得回去。

---

## 风险与控制

### 风险 1：用户一句话就把系统带偏

控制：

- 引入反馈判定 Agent
- 引入学习门禁
- 不允许直接进入强经验

### 风险 2：Agent 为了显得聪明乱回复

控制：

- 输出必须结构化
- `user_reply` 只是展示层，不是事实来源
- 最终动作由状态机和门禁层决定

### 风险 3：项目特例被误学成全局经验

控制：

- 增加 `record_only`
- 单独记录项目私有模式
- 经验聚合时区分项目内和全局

### 风险 4：交互太重，用户嫌麻烦

控制：

- 第一轮由 Agent 先判断
- 只有真的缺关键信息才追问
- 尽量减少追问轮次

### 风险 5：系统看起来会回复，但实际不闭环

控制：

- 会话状态和学习状态拆开
- 每条线程必须有明确落点
- 支持 `closed`、`record_only`、`needs_human_review` 等收口状态

---

## 分阶段落地建议

### 阶段 1：先做“会话化”

目标：

- 从备注框升级成反馈会话抽屉
- 支持用户发起 + Agent 首轮自动回复
- 保存消息和状态

这一步先解决“不是留言板”。

### 阶段 2：补“Agent 追问与结构化判定”

目标：

- Agent 能多轮追问
- 能产出结构化 `FeedbackAgentDecision`
- 前端展示状态流转

这一步解决“不是假聊天”。

### 阶段 3：补“学习门禁”

目标：

- 新增 `accepted_for_learning / record_only / needs_human_review`
- 把 `feedback_samples` 改成吃结构化记录

这一步解决“不是乱学习”。

### 阶段 4：接入运行时经验层

目标：

- 把被接受的经验聚合成运行时提示
- 继续用柔性干预，不直接硬写规则

这一步解决“真的形成闭环”。

---

## 推荐方案

如果只选一个落地方向，我推荐：

**先做“会话式闭环”，不要直接上“全自治学习系统”。**

也就是：

- 先把反馈对象改成会话
- 先让 Agent 负责处理用户反馈
- 先把记录、判定、门禁做对
- 学习层先保守，不激进自动化

原因很简单：

- 这能最快解决你现在最不满意的点：它只是个留言框
- 这也能保住后面继续扩成自循环系统的根
- 同时不会一下子把经验库做脏

---

## 验收信号

这套设计落地后，至少要满足这些外显信号：

1. 用户点击“误报”后，看到的是一条反馈会话，不是一次性备注框
2. Agent 会基于当前问题上下文给出首轮回复，而不是静默收下
3. Agent 在必要时能追问用户，而不是只能收一次输入
4. 每条反馈都能稳定落到：确认误报 / 不确认 / 只记录 / 进入学习 / 转人工 之一
5. 经验层消费的是结构化学习记录，而不是原始留言
6. 后面排查时，能完整回放“用户说了什么、Agent 怎么判断、为什么学习或不学习”

---

## 最后的判断

这次升级的重点不是“给误报按钮加个 AI 回复”，而是：

**把误报反馈从一个静态表单，升级成一个带状态机、带裁判、带门禁、带审计的反馈处理系统。**

只有这样，它才会从“留言界面”变成“自循环系统”的入口。
