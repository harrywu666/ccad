# 审图系统桌面化与工作台演进设计

## 目标

把当前偏“网页业务系统”的前端，升级成一个更像 Mac 桌面工具的审图工作台。

这次不是简单给现有网页套一个桌面壳，而是同时完成两件事：

1. 用 `Electron` 把现有前端包成桌面应用  
2. 借这个机会，把界面升级成真正的“审图工作台”

这个工作台要覆盖整条链路，而不是只覆盖“启动审核之后”的等待阶段：

- 项目管理
- 目录上传与确认
- 图纸管理
- 审图运行
- 报告查看与误报反馈

## 为什么现在要这么做

当前前端已经能跑，但它更像一个传统后台：

- 首页是项目列表
- 点进项目后是详情页
- 审图过程主要靠弹窗承载
- 日志、报告、图纸管理分散在不同区块

这套方式能工作，但不适合长时间、重过程、强本地能力的审图场景。

用户的真实使用感受更像是在“操作一个审图工具”，不是“浏览一个管理页面”。

所以这次的核心目标不是换技术名词，而是把产品形态从：

- `网页管理系统`

升级成：

- `桌面审图工作台`

## 为什么选 Electron

当前阶段更适合 `Electron`，而不是 `Tauri` 或直接重写成 Swift 原生。

原因很简单：

1. 现有前端就是 React/TypeScript，复用成本最低
2. 本地能力要求很重：
   - 本地文件
   - 本地后端
   - 本地 AI Runner / SDK / 登录态
   - 长时间流式输出
3. 现在系统还在快速变化，先选更容易打通的路线更稳

一句大白话：

`Electron` 现在更像一台“先把所有工具装进去能开工的车”，  
`Tauri` 更适合以后流程稳定了再考虑轻量化。

## 设计原则

### 1. 当前审图会话优先

工作台默认不是先看项目管理，而是先看“当前正在处理的会话”。

这意味着：

- 左边主列表是审图会话
- 当前选中的会话决定中间主工作区
- 用户一进入就知道“现在这轮审图做到哪了”

### 2. 项目仍然是顶层归属

虽然主列表是会话，但每条会话都属于一个项目。

不能把“项目”和“会话”做成两个平行系统。

应该这样理解：

- 项目 = 容器
- 审图会话 = 这个项目的一次或多次工作轮次

### 3. 过程常驻，不再靠弹窗承载

“等待弹窗”这种模式要退出主舞台。

工作台里：

- 过程流是固定区域
- Agent 状态是固定区域
- 报告和问题结果是固定区域

不是点一下按钮再弹一个层，把整个长流程塞进去。

### 4. 普通用户看人话，排查时再看原始流

桌面工作台要明确分层：

- 默认层：用户能看懂的进度和结果
- 调试层：AI 原始流、Runner 事件、重试信息

不能把底层原始流直接塞给用户。

### 5. 新工作台与现有 Web 页面共存

第一阶段不会直接替换现有页面。

保留两套入口：

- 现有 Web 详情页
- 新工作台

这样你可以真实比较哪种更顺手，再决定收掉哪一套。

## 整体形态

第一阶段桌面版建议做成三栏工作区。

### 左栏：会话与项目导航

主要放：

- 审图会话列表
- 会话状态
  - 运行中
  - 已完成
  - 已失败
  - 待继续
- 会话所属项目
- 会话最近更新时间

左栏顶部再补一个轻量项目切换器：

- 当前项目
- 项目分类
- 快速新建项目

这样不会丢掉“项目管理”，但主重心仍然是“当前会话”。

### 中栏：主工作区

中栏是用户真正工作的地方。

它不是单一页面，而是一个分区工作区，围绕当前会话展开。

建议分 5 个主视图：

1. `总览`
   - 项目基本信息
   - 当前审图状态
   - 关键统计
   - 最近问题摘要

2. `目录`
   - 上传目录图
   - 目录识别结果
   - 目录确认和编辑

3. `图纸`
   - PDF / DWG 上传
   - 三线匹配状态
   - 图纸列表
   - 图纸预览与状态

4. `审图`
   - 当前阶段
   - 当前 Agent
   - 阶段进度
   - 关键输出

5. `报告`
   - 问题列表
   - 图纸预览
   - Finding 状态
   - 误报反馈
   - 导出报告

一句话：

中栏不是“一个大页面”，而是“围绕当前会话的工作区”。

### 右栏：过程与细节

右栏作为常驻侧栏，放“现在正在发生什么”。

分成两个层次：

1. `进度流`
   - 当前阶段
   - 当前 Agent
   - 关键动作
   - 人话进展

2. `调试流`
   - Runner 事件
   - provider 流
   - 重试
   - 修复
   - 心跳

默认只开第一层。  
第二层只在需要排查时展开。

## 关键页面与用户路径

### 路径 1：新建项目 -> 上传 -> 开始审图

1. 用户进入工作台
2. 在左栏切到某个项目，或新建项目
3. 中栏默认停在 `总览`
4. 引导用户先去 `目录`
5. 再去 `图纸`
6. 数据齐后，从 `审图` 视图直接启动
7. 右栏开始持续显示过程
8. 完成后自动切到 `报告`

### 路径 2：继续看历史会话

1. 左栏切到旧会话
2. 中栏自动展示该会话的总览与报告
3. 右栏展示该会话的历史关键流

### 路径 3：运行中观察

1. 用户停留在 `审图` 视图
2. 中栏看到当前阶段和当前 Agent
3. 右栏持续滚动过程
4. 如果中途失败，右栏和中栏同时提示
5. 用户可切到 `报告` 看已经产出的部分结果

## 与现有页面的对应关系

### 现有可复用的部分

当前这些内容不用推翻：

- 项目列表
- 项目详情里的基础数据加载逻辑
- 目录表格
- 匹配表格
- 报告列表
- 现有审图事件接口
- Finding 状态展示

这些都可以成为工作台里的子模块。

### 需要改变的部分

1. 审图不再靠弹窗承载  
2. 右侧日志面板不再只是一个叠加层  
3. 项目详情不再是唯一主入口  
4. 会话视角要提升为一等对象

## 与 CodePilot 的关系

### 借什么

借的是这些方法：

- 桌面壳
- 会话优先
- 常驻侧栏
- 过程流常驻
- 工作区感

### 不借什么

不直接照抄这些东西：

- 通用聊天产品外形
- 通用代码 IDE 语义
- 过重的模型切换和通用助手能力

我们做的是：

- `审图工作台`

不是：

- `通用 AI IDE`

## 桌面层建议

### 第一阶段

用 `Electron` 包现有 React 前端，形成桌面版：

- 独立窗口
- 桌面菜单
- 本地壳层
- 更稳的本地资源访问

### 第二阶段

再在 Electron 主进程里补这些桌面能力：

- 启动时自检本地后端
- 端口检测
- 本地日志目录入口
- 打开项目目录
- 打开报告目录

### 第三阶段

如果后面桌面体验稳定，再考虑：

- 自动更新
- 多窗口
- 更强的本地资源桥接

## 数据与状态组织建议

### 前端新增的核心对象

除了现有 `Project`，前端还应该新增一个明确的 `AuditSessionViewModel`。

这个对象至少统一整理：

- session_id / audit_version
- project_id / project_name
- status
- current_agent
- current_stage
- progress
- started_at / updated_at
- latest_summary

这样左栏才有稳定的数据源，不会所有地方都临时拼。

### 事件流分层

建议统一成三类：

1. `session_summary`
   - 给左栏用

2. `human_progress`
   - 给右栏默认过程流用

3. `debug_stream`
   - 给调试层用

## 不做什么

为了避免第一版做炸，这些事先不做：

1. 不直接重写成 Swift 原生  
2. 不做多窗口复杂布局  
3. 不做通用聊天产品形态  
4. 不把所有设置都塞进桌面壳第一版  
5. 不把“工作台”做成一上来就支持一堆分屏和插件

## 成功标准

第一阶段完成后，至少应该做到：

1. 用户可以在桌面版里完成从项目到报告的整条主流程  
2. 审图过程不再依赖弹窗  
3. 左栏能明确看到审图会话  
4. 中栏能在目录 / 图纸 / 审图 / 报告之间切换  
5. 右栏能常驻显示过程流  
6. 整体体感更像“桌面工具”，而不是“网页后台”

## 推荐落地顺序

### 第一批

- 先加 Electron 壳
- 跑通现有前端
- 保持旧 Web 页面可继续用

### 第二批

- 新增工作台路由与主框架
- 先做三栏骨架
- 先接当前会话、当前阶段、右侧过程流

### 第三批

- 把目录、图纸、报告模块逐步搬进中栏工作区
- 形成完整闭环

### 第四批

- 再补桌面增强能力

## 最终判断

这次桌面化不应该被理解成“把网页搬到桌面”。

更准确的说法是：

**借 Electron 做壳，借 CodePilot 学方法，把现有审图系统升级成一个真正的桌面审图工作台。**
