# AI 审核系统下一步演进建议

**背景**

当前 AI 审核系统已经完成第一轮架构升级，核心方向是：

- 总控规划Agent
- 关系审查Agent / 索引审查Agent / 尺寸审查Agent / 材料审查Agent
- 证据规划器
- 证据服务层
- 技能包中心
- 误报经验库

这套架构已经比原来的“固定流程 + 固定图片套餐”进步很多，但离“更聪明、更省图、更可控”的目标还有一段距离。下面这份建议，专门针对下一阶段最值得推进的 5 个方向。

---

## 总体判断

这 5 个方案里：

- **直接采纳**：方案三、方案一
- **改造后采纳**：方案五、方案四、方案二

建议的实施顺序：

1. 方案三：审查结果强结构化
2. 方案一：证据规划器改为渐进式
3. 方案五：总控调度器加分层预算
4. 方案四：误报经验库改为放权式接口
5. 方案二：缩小版共享可疑图机制

原因很简单：

- 先把“结果长什么样”定清楚
- 再把“证据怎么逐步加深”定清楚
- 再控制“能花多少视觉预算”
- 再让经验库变得更柔和、更可升级
- 最后再让多个 Agent 之间共享少量高价值信号

这样每一步都能接到前一步，不会出现“模块做出来了，但没有实际挂点”的情况。

---

## 方案三：审查结果强结构化

**结论：优先级最高，直接采纳。**

### 为什么最值得先做

现在系统虽然已经有多个 Agent，但很多结果本质上还是“AI 写了一段描述”。这会带来几个问题：

- 汇总器不好判断这条结果到底算“已确认”还是“待复核”
- 误报经验库很难精确写回
- `legacy / v2` 很难做语义兼容对比
- 前端只能展示文本，没法稳定展示结构状态

所以，下一步最该做的不是再加 Agent，而是把 **Agent 的输出格式先钉死**。

### 建议的数据结构

```python
class Finding(BaseModel):
    sheet_no: str
    location: str
    rule_id: str
    finding_type: Literal["missing_ref", "dim_mismatch", "material_conflict"]
    severity: Literal["error", "warning", "info"]
    status: Literal["confirmed", "suspected", "needs_review"]
    confidence: float
    source_agent: str
    evidence_pack_id: str
    review_round: int
    triggered_by: str | None
    description: str
```

### 必须补的两个字段

#### `review_round`

这个字段非常关键。

同样一条问题：

- 第 1 轮轻量证据下发现
- 第 2 轮补图后发现

它们的意义完全不同。不能只靠一个 `confidence` 数值混在一起。

建议约定：

- `1` = 首轮轻量检查
- `2` = 补图后复核
- `3` = 二次复核

#### `triggered_by`

这个字段用来记录“为什么进入下一轮”。

建议值：

- `confidence_low`
- `target_missing`
- `cross_agent_signal`
- `false_positive_risk`
- `user_feedback_rule`

### 对当前系统的实际帮助

- 汇总器能知道是不是还要补图
- 误报经验库能精确到 `rule_id + source_agent`
- 日志能说清楚“这是首轮怀疑，还是补图后确认”
- 前端以后能直接显示“疑似 / 已确认 / 待复核”

### 不做什么

- 不要求第一阶段就把所有旧文本结果彻底删除
- 不让模型自由新增未定义字段
- 不把 `description` 当成主判断依据，主判断必须来自结构字段

### 落地建议

优先改：

- `cad-review-backend/services/audit/*.py`
- `cad-review-backend/services/audit_runtime/*`
- `cad-review-backend/models.py`
- `cad-review-frontend/src/types/api.ts`

先做兼容层，不要一口气把旧结果格式全删掉。

### 验收信号

- 同一项目跑 `legacy` 和 `v2`，`Finding` 对象都能稳定序列化
- 前端可以直接渲染 `status` 标签，不再靠文本猜状态
- 结果校验环节可以基于字段做机器校验，而不是再额外读描述文本

## 方案一：证据规划器改为渐进式

**结论：直接采纳，但要加上硬边界。**

### 为什么值得做

这件事本质上是在把“先把很多图丢给 AI”改成“先少看，必要时再补图”。

这和现在系统要走的方向完全一致：

- 更省图
- 更省视觉 token
- 更适合复杂图纸
- 对低置信度疑点能看得更细

### 建议的两轮模式

```python
plan = evidence_planner.plan_lite(toc_summary)

if result.confidence < threshold:
    plan.extend(evidence_planner.plan_deep(result.sheet_no))
```

### 我建议补的约束

为了避免系统进入“无限补图”，必须加 3 个边界：

1. 每个任务最多补证据 `1-2` 次
2. 每次补图都必须带原因
3. 证据包必须按级别升级，不能直接乱跳

建议升级链路：

- `overview_pack`
- `focus_pack`
- `deep_pack`

### 不做什么

- 渐进式补图最多两轮
- 第三次仍需要补图时，直接标记 `needs_review`
- 不继续无限取证
- 不允许跳过轻量证据直接默认上最重证据包

### 和方案三的关系

这个方案必须建立在结构化结果之上。

因为只有在结果里有：

- `status`
- `confidence`
- `review_round`
- `triggered_by`

证据规划器才知道要不要发第二轮证据。

### 落地建议

先从：

- `关系审查Agent`

开始试点。因为这块最容易看到“固定大图套餐”带来的浪费。

### 验收信号

- 在 `40` 张图左右的项目里，第二轮补图触发次数低于总任务数的 `30%`
- 同一任务的补图原因能在日志里看见
- 出现第三次补图需求时，系统会稳定落到 `needs_review`，而不是继续扩图

## 方案五：总控调度器加分层预算

**结论：采纳，但不要只做一个全局 token 池。**

### 为什么要做

现在系统虽然已经有并发控制，但还缺“整轮审图到底允许花多少视觉预算”的概念。

没有预算层，就会出现两种坏情况：

- 前面低价值任务吃掉太多图
- 后面真正高风险的图反而看不仔细

### 建议从“单一总预算”升级成“分层预算”

不要只做一个：

- `total_tokens`

而应该至少拆成：

- `image_budget`
- `request_budget`
- `retry_budget`

再加一层：

- `priority_reserve_budget`

也就是“高优先级保底预算”。

### 推荐模型

```python
class VisualBudget:
    global_remaining: int
    reserve_remaining: int
```

预算使用逻辑：

- 普通任务先走全局预算
- 高风险任务允许动用保底预算
- 预算不足时自动降级证据包

例如：

- `deep_pack -> focus_pack`
- `focus_pack -> overview_pack`

### 不做什么

- 预算层只约束证据取用
- 不介入 prompt 构造
- 不直接干预模型调用频率
- 不替代并发控制和重试控制

### 对系统的帮助

- 让系统更可控
- 避免“前面看太多，后面没钱看”
- 能把证据规划器和总控调度器真正串起来

### 落地建议

先只在 `v2` 路径启用，不影响 legacy。

### 验收信号

- 单次审核全程的 `image_budget / request_budget / retry_budget` 消耗都有日志可追踪
- 预算不足时会稳定触发证据包降级，而不是静默失败
- 高优先级任务在全局预算吃紧时仍能动用保底预算完成关键复核

## 方案四：误报经验库改为放权式接口

**结论：采纳，但默认以建议为主，不要一开始就太硬。**

### 为什么值得改

现在误报经验库更像“把规则直接压给审查器”。这样短期有效，但长期有风险：

- 模型升级后，旧经验可能反而限制它
- 经验库会越来越像一堆僵硬的补丁
- 审查器对经验的使用方式不透明

更好的方式是：

- 经验库给“倾向参数”
- 审查器自己决定怎么吸收

### 建议接口

```python
class ExperienceHint(BaseModel):
    rule_id: str
    false_positive_rate: float
    confidence_floor: float
    intervention_level: Literal["hard", "soft", "advisory"]
    reason_template: str | None
```

### 三档干预建议

- `advisory`
  - 仅建议
- `soft`
  - 降权或要求补图
- `hard`
  - 强制跳过或强制压制

### 推荐默认值

- 默认优先 `advisory`
- 少数稳定场景才允许 `soft`
- `hard` 只能用于用户长期确认过的稳定误报模式

### `reason_template` 的价值

不是只告诉 Agent 一个分数，而是告诉它“为什么会这样”。

例如：

- “这类立面索引常因分图号识别被误报”
- “该类材料编号在本项目常有别名写法”

这样 Agent 才知道这条经验能不能迁移到相近场景，而不是死套。

### 落地建议

经验库不要直接控制最终结论，而应影响：

- 任务优先级
- 补图倾向
- 最低置信度门槛
- 结果状态（比如 `suspected` 而不是直接 `confirmed`）

### 不做什么

- 不允许经验库直接改写审查器最终输出
- 不让 `hard` 成为默认干预级别
- 不把没有用户确认的临时样本直接提升成强规则

### 验收信号

- 同一条经验可以切换 `advisory / soft / hard`，而不需要改审查器代码
- 审查器日志里能看到“参考了哪条经验提示”，但结论仍由审查器自己产出
- 模型升级后，只调 `intervention_level` 就能降低经验库干预强度

## 方案二：缩小版共享可疑图机制

**结论：方向对，但不要做成通用消息总线。**

### 为什么不能直接做通用总线

“轻量消息总线”听起来很好，但很容易失控。

一旦三个审查 Agent 都能往里写各种消息，后面会变成：

- 谁都能发
- 谁都在订阅
- 谁都影响谁

最后系统会很难维护。

### 建议缩小成共享可疑图注册表

不要叫 `FindingBus`，建议叫：

- `HotSheetRegistry`
或
- `SharedFindingRegistry`

它只做一件事：

**共享哪张图值得优先看。**

### 先只共享这几类信息

- `sheet_no`
- `finding_type`
- `confidence`
- `source_agent`

### 先只影响一件事

**优先级。**

不要一上来就影响最终判定。

例如：

- 关系审查Agent 发现 `A4.03` 很可疑
- 尺寸审查Agent 优先去看这张
- 材料审查Agent 也提高它的检查优先级

但最终结果还是各自独立判断。

### 这个方案的真正价值

不是“让 Agent 相互聊天”，而是：

**让多个 Agent 对同一张图形成交叉关注。**

这样后面汇总器才能更自然地提升真正高风险问题的可信度。

### 不做什么

- 不做通用消息总线
- 不让共享注册表直接影响最终结论
- 不允许任意 Agent 写入任意类型的自由消息

### 验收信号

- 三个审查 Agent 都能读取共享的可疑图名单
- 共享注册表只改变任务优先级，不直接改变问题结论
- 同一张图被两个 Agent 标记后，汇总器能看到“交叉关注”信号，但原始 Finding 结构保持独立

---

## 推荐的最终实施顺序

### 第 1 步：先做强结构化结果

先把输出对象钉死。  
这是后面所有能力的基础。

### 第 2 步：把证据规划改成渐进式

让“补图”成为一个明确动作，而不是默认行为。

### 第 3 步：给总控接入分层预算

让“补图”和“深看”都在预算约束下进行。

### 第 4 步：把误报经验改成放权式提示

让经验库从“死规则”变成“建议源”。

### 第 5 步：最后再加共享可疑图注册表

先共享高价值可疑图，不做通用消息总线。

---

## 最后结论

下一阶段最重要的，不是继续加更多 Agent，而是把这 4 件事做好：

- Agent 输出更结构化
- 证据获取更渐进
- 视觉预算更可控
- 经验库干预更柔和

如果这 4 件事做好了，你们的系统会从“多 Agent 审图”再往前迈一步，变成：

**一个真正会分层思考、按需拿证据、还能持续纠偏的 Agent 审核系统。**
