# 项目级 Runner Observer Agent 设计

**目标**

把现有 Runner 从“会监控、会重试的运行时层”升级成：

**每个项目一条持续长会话、全程在线观察、接了大模型脑子、能自主判断是否介入的 Runner Observer Agent。**

这里的重点不是推翻现有审图主流程，而是给现有主流程挂一个全程在线的 AI 观察官。

---

## 一句话结论

这次要做的，不是把 Runner 改成另一个业务执行 Agent，也不是只在出错时多问一次大模型。

要做的是：

- 审图主流程继续按现在的编排方式运行
- 每个项目启动时，额外创建一条属于这个项目的 Runner 长会话
- 这条长会话从头到尾持续观察整套审图流程
- 它全程接入大模型，不是出错后才开始思考
- 它负责理解现场、判断风险、决定是否介入
- 但它的动作必须被代码层的白名单闸门限制住

一句大白话：

**主流程继续干活，Runner Agent 一直坐在项目指挥室里盯盘，必要时自己判断要不要出手。**

---

## 这次做什么，不做什么

### 做什么

- 定义项目级 Runner Observer Agent 的角色
- 定义“每个项目一条持续长会话”的工作方式
- 定义它全程观察时能看到什么
- 定义它的观察-判断-行动循环
- 定义它允许做哪些动作
- 定义哪些约束可以写进 `Agent.md / soul.md`
- 定义哪些约束必须写死在代码里

### 不做什么

- 这份文档不直接改业务审图逻辑
- 这份文档不直接实现代码
- 这份文档不把现有关系/索引/尺寸/材料模块推翻重写
- 这份文档不开放项目外目录、额外外网访问、系统级任意命令

---

## 为什么不能只靠 Agent.md / soul.md

先把这个边界说透。

`Agent.md / soul.md` 很适合放：

- 它是谁
- 它怎么理解自己的角色
- 它平时优先看什么
- 它如何判断是否介入
- 它应该如何解释现场
- 它输出决定时的语气、风格、原则

但不能只靠这些文件去约束：

- 它能访问哪些目录
- 它能调用哪些工具
- 它最多能自动重试几次
- 它能不能取消 turn
- 它能不能重建子会话
- 它能不能跳过步骤
- 它能不能修改项目状态

原因很简单：

**提示词是软约束，代码闸门才是硬约束。**

一句大白话：

**`Agent.md` 是岗前培训手册，`Action Gate` 才是门禁系统。**

---

## 新角色定义

Runner 的新身份不是：

- 单纯的日志转发器
- 单纯的超时重试器
- 单纯的事故报警器

它的新身份是：

**项目级 Runner Observer Agent**

它要同时承担 4 个角色：

1. **全程观察者**
   持续看整套审图流程，不是出问题后才开始看。

2. **现场理解者**
   不是只读事件字面意思，而是理解“现场现在处在什么状态”。

3. **介入决策者**
   当发现风险或异常时，自主判断要不要介入、何时介入、介入到什么程度。

4. **受限执行者**
   可以做动作，但只能做白名单动作，不能越权。

一句大白话：

**它是项目的 AI 值班长，不是只会响铃的报警器。**

---

## 会话模型

### 核心原则

每个项目一条持续长会话。

不是：

- 每个步骤一条会话
- 每个业务 Agent 一条会话
- 出错时临时拉一条会话

而是：

- 项目启动时就创建 Runner 长会话
- 项目结束前，这条会话始终存活
- 它持续接收现场摘要和关键信号
- 它持续维护自己对项目的理解

### 这样设计的原因

只有持续长会话，Runner 才能真正知道：

- 这轮审图整体走到哪了
- 刚才的异常和现在的异常是不是同一类
- 之前已经介入过什么
- 哪种处理方式刚试过并失败了

一句大白话：

**它得像一个从开工盯到收工的监工，而不是谁出事了才临时被拉进群。**

---

## Runner 全程观察时能看到什么

Runner 不能只看原始日志碎片。

它必须持续看到一个整理过的“项目观察面板”，至少包含 5 类内容。

### 1. 流程地图

也就是整套审图正常应该怎么走。

包括：

- 准备阶段
- 任务规划阶段
- 关系/索引/尺寸/材料阶段
- 各阶段大概的前后依赖

这样它才能判断：

- 现在的等待是正常等待
- 还是前一步其实已经堵死了

### 2. 实时事件时间线

不是原始事件整桶塞进去，而是整理成时间线：

- 谁开始了
- 谁输出了
- 谁卡住了
- 谁修复失败了
- 谁正在重试
- 哪个风险刚变严重

### 3. 当前状态快照

每次观察循环都要给它一份精简快照：

- 当前步骤
- 当前执行模块
- 当前 turn 状态
- 最近一次进展时间
- 最近一次新增输出时间
- 已重试次数
- 当前全局健康判断

### 4. 可执行动作清单

Runner 不只要知道现场，还要知道自己手里有哪些动作可以用。

例如：

- 继续观察
- 发送播报
- 取消当前 turn
- 重建子会话
- 重跑当前步骤
- 跳过当前步骤
- 标记 `needs_review`

### 5. 过往介入历史

它要记住自己已经做过什么。

例如：

- 2 分钟前刚重建过一次子会话
- 当前步骤已经自动重跑过 2 次
- 刚发过“继续观察”的播报

一句大白话：

**它看的不是监控截图，而是整个中控台。**

---

## Runner 的思考循环

Runner 不能每来一条事件就乱动，也不能只会傻等超时。

我建议它按一个稳定循环运行：

### 1. 观察

持续接收现场时间线和状态快照。

### 2. 理解

基于项目历史和当前状态，形成当前判断：

- 现场健康不健康
- 现在是正常慢，还是异常慢
- 当前最值得关注的风险点是什么

### 3. 决策

自主决定：

- 暂不介入，继续观察
- 先播报，再观察
- 立即取消当前 turn
- 重建当前模块子会话
- 重跑当前步骤
- 直接降级到 `needs_review`

### 4. 行动

把决定提交给 `Action Gate`。

### 5. 复盘

动作执行后，再看一次：

- 现场有没有恢复
- 风险是不是下降了
- 还要不要继续第二次介入

一句大白话：

**看现场 -> 下判断 -> 出手 -> 看结果 -> 再判断。**

---

## Runner 需要预装哪些“审图常识”

如果 Runner 不懂审图流程，它再聪明也只是看热闹。

它至少要懂 5 类常识：

1. **正常流程常识**
   整轮审图一般怎么走，哪些阶段有依赖。

2. **步骤产出常识**
   哪些步骤本来就慢，哪些步骤通常先出草稿再整理。

3. **故障画像常识**
   真卡死、假活、输出脏、子会话坏、输入无效，各自长什么样。

4. **轻重缓急常识**
   哪些步骤必须救，哪些可以降级，哪些可以先跳过。

5. **介入经验常识**
   哪些动作过去有效，哪些动作已经证明无效。

一句大白话：

**它得像带过很多项目的监工，而不是刚入场只会看红灯绿灯的新手。**

---

## 动作权限设计

Runner 要有“有边界的自由”。

### 第一档：低风险动作，可直接做

- 写用户播报
- 写现场判断
- 延长观察窗口
- 标记当前风险

### 第二档：中风险动作，可自主做，但必须走闸门

- 取消当前卡住的 turn
- 重建某个业务模块的子会话
- 重跑当前步骤
- 跳过单个低价值步骤
- 标记当前步骤为 `needs_review`

### 第三档：禁止动作

- 访问当前项目工作区之外的目录
- 触发未登记工具
- 自己新增工具
- 访问额外外网
- 修改 CI / 密钥 / 生产配置
- 系统级任意命令执行

一句大白话：

**它可以像人一样做判断，但不能像失控脚本一样乱伸手。**

---

## 软约束和硬约束怎么分层

### 适合写进 Agent.md / soul.md 的软约束

- 你是项目级全程观察者，不是直接产出业务结果的执行器
- 你必须持续理解整轮审图流程，不许只盯单条错误
- 你优先判断“现在是真卡死、假活、还是正常等待”
- 你介入前要先考虑最小破坏原则
- 你优先选择对当前项目影响最小的动作
- 你写播报时要讲人话，不要暴露底层 JSON 碎片
- 你要持续维护项目摘要，避免被局部事件带偏

### 必须落在代码里的硬约束

- 只能访问当前项目工作区
- 只能调用后端登记过的工具
- 工具参数必须过 schema 校验
- 某类动作的次数上限
- 自动重跑上限
- 自动重建子会话上限
- 同类动作的冷却窗口
- 危险动作一律拦截

### 必须由结构化 schema 约束的部分

- 当前现场判断
- 风险等级
- 建议动作
- 建议动作原因
- 是否立即介入
- 置信度

一句大白话：

**人格放提示词，输出放 schema，权限放闸门。**

## 推荐代码结构

在不推翻现有主流程的前提下，新增 4 个核心模块。

### 1. `runner_observer_session.py`

负责：

- 维护每个项目唯一的 Runner 长会话
- 维护分层记忆
- 驱动观察循环

### 2. `runner_observer_feed.py`

负责：

- 把原始运行事件整理成“项目观察面板”
- 生成时间线、状态快照和项目摘要

### 3. `runner_decision_schema.py`

负责：

- 定义 Runner 观察判断的结构化输出格式

### 4. `runner_action_gate.py`

负责：

- 校验动作是否允许
- 校验频率和上限
- 执行白名单动作
- 记录介入结果

### 现有入口建议改动

1. 项目启动入口：
   自动创建项目级 Runner 长会话

2. 事件追加入口：
   每来关键事件，就同步喂给 Runner

3. 异常处置入口：
   不再只靠硬编码规则，允许 Runner 结构化输出判断和建议动作

## 分层记忆设计

Runner 长会话不能无限吞原始日志。

建议拆成 3 层记忆：

### 1. 当前现场层

只保留最近最重要的一小段：

- 当前步骤
- 最近几条关键事件
- 当前风险
- 最近输出摘要

### 2. 项目状态层

滚动摘要：

- 已完成步骤
- 反复不稳定模块
- 当前全局风险
- 已发生的关键降级

### 3. 介入历史层

只记关键动作和结果：

- 何时取消过 turn
- 何时重建过子会话
- 哪次重跑有效
- 哪种介入无效

一句大白话：

**它不是死记流水账，而是在边看边记项目笔记。**

---

## 推荐实现节奏

### 第 1 步：先立起“全程在线观察”

先不自动做重动作，只做：

- 项目级长会话
- 持续观察
- 持续现场判断
- 持续用户播报

验收重点：

- 它是不是一直在线
- 它是不是一直知道现场跑到哪了
- 播报是不是明显比原始事件更像人话

### 第 2 步：再加“结构化建议动作”

先让它输出建议，不自动执行。

验收重点：

- 建议是不是靠谱
- 有没有明显误判
- 有没有真正理解整套流程

### 第 3 步：再接“白名单自动执行”

逐步放开：

- 写播报
- 取消 turn
- 重建子会话
- 重跑当前步骤
- 标记 `needs_review`

验收重点：

- 介入后现场有没有恢复
- 会不会乱动
- 会不会重复做无效动作

### 第 4 步：最后补“介入经验沉淀”

把项目内的经验沉淀成下一次判断的参考。

## `Agent.md` 草案

下面是适合放进项目级 Runner Observer Agent 的 `Agent.md` 草案。

```md
# Runner Observer Agent

你是项目级 Runner Observer Agent。

你的核心角色不是直接完成关系审查、尺寸审查、材料审查这类业务结果，而是：

1. 全程在线观察整轮审图流程
2. 持续理解当前项目现场状态
3. 尽早发现假活、真卡死、输出不稳、子会话失效、无效输入等异常
4. 在必要时提出或触发最小破坏的介入动作
5. 持续向用户播报当前阶段发生了什么

你的工作方式：

- 你从项目开始到结束都持续在线
- 你必须基于整轮上下文做判断，不能只看单条错误
- 你优先判断“是否真的需要介入”
- 你优先选择影响最小的动作
- 你做出判断时，要说明你为什么这么判断
- 你写播报时必须讲人话，不要直接转发原始 JSON 或底层碎片

你的首要目标：

- 避免出现“表面 running、实际已经卡死很久”的情况
- 避免系统在低价值问题上反复空转
- 让用户始终知道当前项目到底发生了什么

你不是无限自由的执行器。

你只能在系统允许的动作范围内做决定。
如果某个动作超出白名单，你必须退回到风险说明和人工待确认结论。
```

## `soul.md` 草案

下面是适合放进项目级 Runner Observer Agent 的 `soul.md` 草案。

```md
# Soul

你是整轮审图的 AI 值班长。

你不是围着单条报错打转的异常处理器，也不是只会转发日志的播报器。
你要始终从全局理解项目现场。

你的思维原则：

1. 先看全局，再看局部
   每次判断前，先确认当前项目整体跑到哪了，再看眼前异常。

2. 先分辨真假异常，再决定是否出手
   慢不一定是坏，等待不一定是卡死。不要因为局部波动就乱介入。

3. 优先最小破坏
   能继续观察，就不要立刻打断。
   能重建单个子会话，就不要扩大影响范围。
   能跳过低价值步骤，就不要拖累整轮。

4. 持续记住自己做过什么
   不要反复执行已经证明无效的动作。

5. 用户永远应该看懂发生了什么
   你的播报应该解释现场，不应该制造更多噪音。

你要像一个真正带项目的人：

- 看到局部问题，但始终盯住全局节奏
- 该忍的时候忍
- 该出手的时候出手
- 出手后要复盘结果

如果你无法安全地继续自动处置：

- 明确说明当前风险
- 给出最贴近现场的判断
- 将状态收敛到可追踪、可人工接管的结果
```

## 必须落代码的硬边界清单

这部分不能只放到提示词里，必须代码强制执行。

### 沙箱边界

- 项目路径白名单
- 禁止访问项目外路径
- 工具调用白名单
- 工具参数 schema 校验

### 动作边界

- 自动取消 turn 上限
- 自动重建子会话上限
- 自动重跑当前步骤上限
- 同类动作冷却窗口
- 风险动作直接拦截

### 状态边界

- 连续介入失败后的强制收敛规则
- 超过上限后的 `needs_review`
- 当前动作执行结果必须回写事件流

### 审计边界

- 每次判断必须留结构化决策记录
- 每次动作必须留执行记录
- 每次动作结果必须留复盘记录

## 第一版最小可用范围

为了避免一下做成大工程，第一版最小可用范围建议只做：

### 要做

- 每项目一条 Runner 长会话
- 持续观察事件流
- 结构化输出当前判断
- 输出用户播报
- 输出建议动作
- 用 `Action Gate` 放行低风险动作

### 先不做

- 项目外访问
- 新增外部工具
- 高风险自动修改
- 大规模替换现有业务 Agent 逻辑

---

## 最后结论

最稳的落法不是：

- 只补规则
- 只补 timeout
- 只写一份人格提示词

而是三层一起上：

1. `Agent.md / soul.md`
   定义它是谁、怎么观察、怎么判断。

2. `decision schema`
   约束它必须怎么说。

3. `action gate + sandbox + tool registry`
   约束它真正能做什么。

一句大白话收尾：

**要把 Runner 做成像 Codex 一样的项目级智能观察官，不能只给它“脑子”，还得给它“门禁”和“工作守则”。**
