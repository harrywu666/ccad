# 审图真流式输出与断线重连设计

**目标：** 让 Kimi 在审图过程中的返回内容以真实流式方式进入后端，再从后端继续实时推到前端。这样既能看到 AI 一步步吐出的内容，也能在中途断开时自动续上，而不是整次请求白等。

## 现状

当前系统已经有：

- 后端事件表 `audit_run_events`
- 查询接口 `/api/projects/{project_id}/audit/events`
- 前端 Agent 视角日志面板

但它还不是真正的“AI 流式输出”，主要问题有 4 个：

1. `call_kimi()` 还是一次性等完整响应，没有接 Kimi 的官方流式输出。
2. 后端拿不到 Kimi 中间吐出的内容，所以看不到它是怎么想、怎么规划、断在哪一步的。
3. 断线后现在只能整次重试，不能优雅续跑。
4. 前端现在看到的只是“后处理后的事件”，不是 Kimi 的真实过程。

## 方案对比

### 方案 A：继续只用一次性请求 + 轮询事件

优点：

- 最省改动
- 复用现有接口最多

缺点：

- 看不到 Kimi 中间过程
- 断了就只能整次失败
- 无法判断 plan 阶段到底卡在哪里
- 后面还得继续改

**结论：** 只能临时顶着，不适合继续扩。

### 方案 B：只做前端 SSE / WebSocket，不改 Kimi 上游调用

优点：

- 前端看起来更实时

缺点：

- 只是“界面更像流式”，不是真流式
- 后端依旧拿不到 Kimi 中间内容
- 断线后上游还是整次失败
- 解决不了你最关心的“它为什么断、断前走到哪”

**结论：** 方向不对，只改前端没意义。

### 方案 C：Kimi 上游真流式 + 后端事件落盘 + 前端 SSE 展示

优点：

- 后端能拿到 Kimi 真正的中间输出
- 能看到 plan 阶段和审图阶段到底在想什么
- 断线时可以按阶段重试、按事件续上
- 前后端看到的是同一条真实流
- 继续保留现有事件表和 `since_id`

缺点：

- 实现会比单纯做前端轮询复杂
- 需要重构 `kimi_service.py`

**结论：** 这是当前最合适的方案。

## 推荐方案

采用 **Kimi 上游真流式 + 后端事件表落盘 + 前端 SSE 展示 + 轮询兜底**。

大白话就是：

- Kimi 一边生成，后端一边接收。
- 后端把这条流拆成“过程片段”和“阶段事件”两类信息，持续写入事件表。
- 前端订阅这条事件流，所以你看到的是 AI 的真实运行过程，不是事后拼出来的卡片。
- 如果上游或下游中途断了，都要尽量自动续上。

## 核心设计

### 1. 后端保留“事件表”作为唯一事实来源

不直接把 stdout 推给前端，也不直接把 Kimi 原始 socket 数据裸透出去。

还是继续以 `audit_run_events` 为准，因为它有几个好处：

- 刷新页面后还能补历史
- 断线后可以从上次编号继续拉
- 后面人工验收、失败排查也能复用

### 2. `call_kimi` 拆成“一次性调用”和“流式调用”

新增两类入口：

- `call_kimi()`：保留一次性兼容路径
- `call_kimi_stream()`：接 Kimi 官方流式输出

其中 `call_kimi_stream()` 的职责是：

- 与 Kimi 建立流式连接
- 一段一段接收返回内容
- 每收到一段，就把它送到“运行事件写入器”
- 最后再把完整内容组装出来给业务模块解析

先接入的阶段：

- `总控规划Agent`
- `关系审查Agent`
- `索引审查Agent` 的 AI 复核

### 3. 后端要把“模型流片段”和“阶段事件”分开写

建议事件至少分两类：

- `model_stream_delta`
  - Kimi 刚吐出来的一小段内容
  - 用于观察模型实时过程
- `phase_event`
  - 阶段开始、完成、补图、重试、失败
  - 用于给普通用户看清大流程

这样好处是：

- 你可以看 AI 正在输出什么
- 普通用户也能看懂系统现在做到哪一步
- 后端排查时能知道是“模型没回”“模型回了脏内容”“还是中间断了”

### 4. 前端以 SSE 为主，轮询为备

前端逻辑分三层：

1. 先尝试 SSE
2. SSE 断开后自动重连
3. 多次重连失败后，自动退回轮询

恢复时带上：

- `lastEventId`
- 或本地维护的 `since_id`

这样断开后只补没收到的那一段。

### 5. 重连要分上下两层

**心跳间隔统一约定：`25` 秒。**

原因很简单：

- 太短会刷太多无意义事件
- 太长又容易让代理层或网关误判成“连接已经没了”

具体规则：

- `model_stream_delta` 超过 `30` 秒无新增时，后端补一条心跳事件
- 后端 SSE 通道每 `25` 秒发一次心跳
- 这样可以尽量避开常见网关默认 `60` 秒超时断连

这个值先固定，不让实现时各自拍脑袋。

#### 上游重连：Kimi -> 后端

分 3 类情况：

1. **服务忙**
   - 如 `429`
   - 自动等待再重试
   - 把“本次等待重试”写进事件流

2. **临时故障**
   - 如 `500/502/503/504`
   - 自动退避重试
   - 保留已接收的流片段

3. **长时间无进展**
   - 写心跳事件
   - 到达上限后标记当前阶段失败

#### 下游重连：后端 -> 前端

重连分 3 类情况：

1. **短断线**
   - 直接自动重连
   - 用 `lastEventId` 续上

2. **长断线**
   - 提示“连接刚刚中断，正在重连”
   - 重连成功后补齐中间漏掉的事件

3. **持续失败**
   - 自动切到轮询
   - 页面提示“已切换到兼容模式，日志可能有一点延迟”

### 6. 日志窗口展示两层内容

前端不要直接把所有模型流片段都像聊天记录一样刷满页面。

建议做两层：

1. **默认层**
   - 给普通用户看
   - 展示大白话阶段事件

2. **过程层**
   - 给排查和观察用
   - 展示 Kimi 流式片段
   - 重点看 plan 阶段、关系分析阶段、AI 复核阶段

## 不做什么

- 这一步不直接上 WebSocket
- 这一步不做“只有前端在演流式”的假方案
- 这一步不要求所有 Agent 一次性都改成流式
- 这一步不直接把 Kimi 原始全文无脑展示给普通用户
- 这一步不改动当前事件表作为主数据源的设计

## 验收信号

1. plan 阶段能看到 Kimi 真实流式输出，而不是只看到最后成败。
2. 关系分析阶段如果断开，后端能写出“断在什么阶段、已重试几次”。
3. 断开浏览器网络再恢复后，前端日志能自动续上，不丢中间段。
4. SSE 持续失败时，前端能自动退回轮询，页面仍能继续看日志。
5. 等待界面里能明确告诉用户：
   - 当前是谁在工作
   - 刚刚做了什么
   - Kimi 是否正在持续输出
   - 连接是否正常
