# 子 Agent 精确定位锁定设计

**目标**

把“问题成立”的标准前移到子 Agent 运行阶段：子 Agent 输出的每一条问题，都必须同时带上图纸中的精确定位和可直接画云线的区域。不是后面再猜位置，不是前端再补救。

**使用的 skills**

- `brainstorming`
- `writing-plans`

---

## 1. 背景

当前系统已经有三样基础能力：

1. 子 Agent 能发现问题
2. 后端能保存锚点、坐标和高亮框
3. 前端能在右侧图纸预览里画云线

但真正的问题在于：

- 子 Agent 现在还可以在“没给出精确定位”的情况下提交问题
- 后端预览层会用回退方式硬凑一个位置
- 前端会把“估计位置”也展示给用户

这会导致三个后果：

1. 问题描述可能是错的，但仍然入库
2. 右侧画出的云线可能圈错地方
3. 一条问题到底成不成立，标准变得很模糊

这和审图的本质冲突。审图不是“模型觉得哪里不对”，而是“图纸上能指出具体哪里不对”。

---

## 2. 核心原则

### 原则 1：问题和定位是一体的

以后子 Agent 不能只输出“问题内容”，必须同时输出：

- 问题文本
- 对应图纸
- 精确坐标
- 可画云线的高亮框
- 定位置信度
- 该定位对应的证据对象

少一个，都不算一条完成的问题。

### 原则 2：没定位成功，不是“问题不存在”，而是“子 Agent 还没交完卷”

如果子 Agent 第一次没能定位成功：

- 这条问题不能直接入正式结果
- 也不能丢给预览接口去猜
- 而应该在子 Agent 内部继续重试定位

大白话就是：真问题不能因为系统不会圈线就消失，但系统也不能在没圈线时假装已经搞定了。

### 原则 3：总控和 Runner 负责救子 Agent，不负责替子 Agent 做定位

职责要锁死：

- 子 Agent：发现问题 + 给出精确定位
- 总控 Agent：子 Agent 定位失败时，带记忆重启子 Agent
- Runner Agent：总控救不动时，再介入协调

不要再让后处理层变成“帮子 Agent 擦屁股”的兜底定位器。

### 原则 4：用户看到的每条问题，都必须能在图上被指出来

这不是展示要求，而是事实要求。

系统标准应该变成：

> 只有“问题内容成立”并且“图纸位置成立”的问题，才算正式问题。

---

## 3. 新的结果生命周期

把现在“发现问题 -> 直接入结果”的流程，改成下面这条：

1. `draft_found`
   子 Agent 先发现一个候选问题

2. `grounding_in_progress`
   子 Agent 对这个候选问题做精确定位

3. `grounded`
   子 Agent 已拿到：
   - 锚点
   - `highlight_region`
   - 证据对象
   - 置信度

4. `committable`
   通过输出校验，可以正式写入 `audit_results`

5. `grounding_failed`
   子 Agent 多次定位失败，交给总控判断是否重跑子 Agent

重点：

- `grounded` 之前不能进正式结果
- `grounding_failed` 不等于问题被删除
- 只是说明子 Agent 还没把“图上哪里错”说清楚

---

## 4. 子 Agent 输出合同

## 4.1 通用字段

每条问题都必须带：

- `sheet_no`
- `issue_type`
- `description`
- `location`
- `anchor`
- `highlight_region`
- `confidence`
- `evidence_refs`

其中：

- `anchor` 至少要有 `global_pct`
- `highlight_region` 至少要有 `bbox_pct`
- `confidence` 不能缺
- `evidence_refs` 要说明这个框是从哪个对象来的

## 4.2 强校验

只要出现下面任何一种情况，这条输出都判定为失败：

- 没 `anchor`
- 没 `highlight_region`
- `highlight_region.width <= 0`
- `highlight_region.height <= 0`
- `anchor.sheet_no` 和问题图号不一致
- 文本描述中的索引号 / 目标图号和证据对象不一致

这不是警告，是直接不让提交。

---

## 5. 各类问题怎么锁定位

## 5.1 索引问题

索引问题最适合先收紧，因为对象天然明确。

每条索引问题必须锁定：

- 源图中的索引圆圈或索引头
- 索引号文本
- 目标图号文本
- 对应局部 bbox

额外增加一步二次核对：

- 从定位框附近重新读一次索引号
- 从定位框附近重新读一次目标图号
- 与最终描述逐字比对

如果描述写的是 `A1 -> G0.04b`，但框里实际读到的是 `A2 / G0.04a`，直接判失败。

## 5.2 尺寸问题

尺寸问题不能再只输出：

- “下部横向尺寸链”
- “第二段墙体”

必须锁定到：

- 具体尺寸对象 ID
- 相关尺寸文本块
- 对应尺寸链局部框

如果一条尺寸问题涉及 A 图和 B 图对比：

- 两边都要有自己的定位框
- 不允许只给一边有框，另一边靠文字描述

## 5.3 材料问题

材料问题必须锁定到：

- 材料文本块
- 引线末端
- 或材料标注关联对象附近的局部框

不能只写“某处材料不一致”。

---

## 6. 一组问题必须有一组云线

如果左侧一条分组问题实际包含多个子问题：

- 每个子问题都必须有独立锚点
- 每个子问题都必须有独立 `highlight_region`
- 前端必须全部画出来

不是“选一个代表点”。

否则用户看到的是：

- 左边一堆问题
- 右边只圈一个位置

这会天然削弱可信度。

---

## 7. 总控和 Runner 的新职责

## 7.1 总控 Agent

总控新增一个判断：

不是只看“子 Agent 有没有跑完”，还要看：

- 子 Agent 提交的问题里，`grounded` 比例够不够
- 是否连续多次卡在 `grounding_failed`
- 是否反复重跑同一批定位任务

如果是：

- 总控带着任务记忆重启子 Agent

## 7.2 Runner Agent

Runner 不直接补定位，但要看：

- 总控有没有持续把“未定位问题”留在那里不处理
- 总控是不是开始原地打转

如果是：

- Runner 介入重启总控

---

## 8. 前端展示规则

前端右侧预览以后只做两件事：

1. 展示子 Agent 已经锁死的定位框
2. 把一组问题的多个框同时画出来

前端不再负责：

- 猜位置
- 代替后端补定位
- 把低置信度位置包装成“像是真的”

换句话说：

前端只负责显示已经成立的图上证据。

---

## 9. 数据落库变化

正式问题写入 `audit_results` 时，要求：

- `evidence_json.anchors` 不能为空
- 每个正式问题至少有一个 `highlight_region`

这里必须先做一个前置确认：

- 先核对当前 `audit_results.evidence_json` 的实际结构
- 确认 `anchors` 现在到底是数组，还是只是“看起来像数组、实际只存一个主锚点”

如果当前结构已经稳定支持数组：

- 继续沿用数组结构
- 每个子问题往 `anchors` 里追加自己的独立锚点

如果当前结构实际上只适合单锚点：

- 必须先升级序列化格式
- 必要时同步补数据迁移或兼容读写逻辑
- 然后再做“分组问题多锚点”能力

也就是说，“一组问题多个云线”不是纯前端问题，它先取决于后端落库格式是否真能装下多个独立锚点。

`audit_issue_drawings` 也要同步存：

- 源图定位框
- 目标图定位框
- 证据对象来源
- 定位置信度

对“分组问题”，要支持多个独立锚点，而不是只留一个主锚点。

如果最终决定沿用 `evidence_json.anchors` 数组作为唯一来源，那么：

- `audit_issue_drawings` 负责一条问题和一张图的关联记录
- `evidence_json.anchors` 负责一条问题内部的多锚点明细

两者职责要分清，不要一处存主锚点，另一处存多锚点，最后互相打架。

---

## 10. 验收标准

这套方案做完后，至少要满足：

1. 索引问题里，图上读到的索引号和问题描述一致
2. 每条正式问题都能在右侧预览中看到明确云线
3. 一条分组问题的多个子问题能同时看到多个云线
4. 尺寸/材料问题不再出现“模棱两可但无框”的正式结果
5. 子 Agent 如果没给出 `highlight_region`，这条问题不能进入正式结果

---

## 11. 推荐实施顺序

1. 先收紧索引问题
2. 再收紧尺寸问题
3. 再收紧材料问题
4. 最后统一前端多云线展示

原因很简单：

- 索引问题对象最明确
- 最容易先把“问题内容 + 位置内容”真正锁成一体

---

## 12. 下一步建议

下一步不要先改前端，也不要先加“展示过滤”。

正确顺序是：

1. 先把子 Agent 的输出合同改掉
2. 再把“没定位就不能提交”钉死在校验层
3. 最后让前端老老实实展示这些已经锁死的框

这才是在修根因，不是在补表面。
