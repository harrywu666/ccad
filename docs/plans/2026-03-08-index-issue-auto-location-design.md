# 索引问题自动定位设计

**日期：** 2026-03-08

## 背景

当前索引类问题已经能在后端产出 `evidence_json.anchors`，其中包含源图索引的 `grid/global_pct`。但前端查看问题时，仍然主要依赖 `sheet_no_a/sheet_no_b` 去挑选图纸，因此存在两个核心问题：

1. 目标图不存在时，问题明明成立，但界面无法直接落到源图的出错位置。
2. 审核结果没有绑定到具体的图纸版本；图纸重传后，旧审核版本可能被错误地展示到新的图纸版本上。

以用户给出的案例为例：

- 源图存在：`A6.00`
- 源图中的 `索引3` 可以定位
- 目标图号写错：`A06.00a`
- 正确行为应为：
  - 问题继续成立，因为 `A06.00a` 不存在
  - 查看问题时直接打开 `A6.00`，自动定位到 `索引3`

## 目标

构建“问题和图纸的精确匹配”链路，使系统在产出索引问题时自动记录：

- 这条问题对应哪张具体图纸
- 是 A 侧还是 B 侧
- 对应哪个索引编号
- 精确坐标在哪里
- 属于哪个审核版本、哪个图纸数据版本

前端查看问题时直接使用这份结构化记录，而不是重新靠 `sheet_no` 猜测。

## 非目标

- 不做人工点选、人工修正、人工兜底流程
- 不自动“纠正”错误图号并吞掉问题
- 不改变索引问题本身的业务判定逻辑

## 设计原则

1. 问题判定与问题定位分离
   - 判定仍然基于原始标注文本是否指向了真实存在的目标图
   - 定位优先基于源图索引锚点
2. 审核版本与图纸版本都要隔离
   - 同一 `audit_version` 的问题定位必须可重放
   - 图纸重新上传后，旧审核结果不能串到新图
3. 自动化优先
   - 所有定位信息来自已有 DWG/JSON 坐标和审核产物
   - 无人工录入步骤

## 方案比较

### 方案一：仅前端临时根据 `sheet_no_a/sheet_no_b` 推断图纸

优点：
- 改动小

缺点：
- 目标图不存在时无法稳定定位
- 图纸版本会串
- 无法形成后端可追溯记录

结论：不采用。

### 方案二：只复用 `audit_results.evidence_json`

优点：
- 复用现有字段

缺点：
- 只有锚点，没有明确绑定到具体 `drawing_id/data_version`
- 图纸重传后旧审核版本仍可能错连
- 前端仍需自己做图纸选择逻辑

结论：不足以满足版本隔离要求。

### 方案三：新增“问题图纸匹配记录”并提供专用预览接口

优点：
- 问题、图纸、坐标、审核版本、图纸版本全部可追溯
- 目标图不存在时仍可稳定落到源图
- 前端逻辑简单，直接消费结构化预览结果

缺点：
- 需要新增表、回填和接口

结论：采用。

## 数据模型

新增表 `audit_issue_drawings`（命名可在实现时微调），建议字段：

- `id`
- `project_id`
- `audit_result_id`
- `audit_version`
- `match_side`：`source` / `target`
- `drawing_id`
- `drawing_data_version`
- `sheet_no`
- `sheet_name`
- `index_no`
- `anchor_json`
- `match_status`：`matched` / `missing_drawing` / `missing_anchor`
- `created_at`
- `updated_at`

约束建议：

- `UNIQUE(audit_result_id, match_side)`

说明：

- 对于“目标图不存在”的问题，只会写入 `source` 侧的 `matched` 记录；`target` 侧可省略，或写一条 `missing_drawing` 记录但不带 `drawing_id`
- `anchor_json` 存储 `grid/global_pct/role/origin/confidence`

## 后端流程

### 1. 审核产出阶段

在索引审核生成 `AuditResult` 后，立即根据 `evidence_json.anchors` 生成问题图纸匹配记录：

- 优先取 `role=source` 的 anchor，绑定到源图
- 若 `role=target` 存在且目标图能解析到具体图纸，则补写 target 侧记录
- 若目标图不存在，问题照常保留，仅不生成 target 侧有效图纸记录

### 2. 图纸版本绑定

匹配记录必须绑定审核当时的具体 `drawing_id + drawing_data_version`：

- 不允许查看时再按最新图纸重新猜
- 图纸被替换后，旧 `drawing_id` 仍保留，可用于历史审核回放

### 3. 预览接口

新增按问题读取的接口，例如：

- `GET /api/projects/{project_id}/audit/results/{result_id}/preview`

返回：

- 当前问题基础信息
- source/target 两侧图纸记录
- 每侧图纸的图片 URL 所需信息
- 自动定位所需的 anchor
- 目标图缺失原因

## 前端交互

### 问题列表 -> 右侧看图

点开问题时：

1. 先请求问题预览接口
2. 如果 source 有匹配图纸，则默认展示 source
3. 如果 target 也有有效图纸，则允许切换 A/B
4. 如果 target 不存在，则仍只展示 A 图，并在说明中明确“目标图不存在”

### 自动定位

画布加载后：

- 根据 anchor 中的 `global_pct` 自动平移/缩放到目标区域
- 叠加系统定位标记
- 系统定位标记与人工标注层分离，不写入 `drawing_annotations`

## 风险与处理

### 1. 图号归一化过宽导致误匹配

处理：
- 索引问题判定仍使用原始 `target_sheet`
- 问题定位只绑定源图锚点，不因推测“正确图号”而吞错

### 2. 旧审核版本串到新图

处理：
- 使用独立匹配表持久化 `drawing_id/data_version`

### 3. 只有 grid 没有 global_pct

处理：
- 接口仍返回 grid
- 前端优先使用 `global_pct`，没有时可降级到文案提示，但不伪造坐标

## 测试策略

重点覆盖：

1. 源图存在、目标图号拼写错误、目标图不存在
2. 目标图存在但缺少对应索引编号
3. 图纸重传后查看旧审核版本，仍命中旧 `drawing_id`
4. 前端打开问题时默认定位到 source anchor
5. target 缺失时 UI 仍可正常查看 A 图

## 结论

采用“索引问题继续报错 + 预览永远优先落到源图索引锚点 + 后端持久化问题图纸匹配记录”的方案。这样既保留了业务正确性，也满足了自动定位与版本隔离要求。
